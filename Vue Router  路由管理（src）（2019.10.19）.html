AMD / CMD + 打包工具

单页面应用模式SPA



应用过程：一个外壳页面和多个页面片段构成



跳转方式：一个页面删除或隐藏，加载另一个页面片段并显示，片段间的模拟跳转，没有开壳



跳转后公共资源是否重新加载：  否



URL模式：http://xxx/snel.html#page1  和  http://xxx/snell.html#page2



用户体验：页面片段间切换快，用户体验好，包括移动设备



能否实现转场动画：容易实现（手机APP动效）



页面间传递数据：页面传递数据容易（Vuex 或 Vue 中的父子组件通讯 props 对象）



搜索引擎优化：需要单独方案（SSR）  



特别适用的范围：对体验要求高，特别是移动应用



开发难度：较高，需要专门的框架来降低这种模式的开发难度

AMD / CMD

AMD需要在模块最开始就把当前模块依赖的内容导入进来，然后再开发

CMD比较灵活，在开发过程中需要用到哪些模块，再单独导入即可

AMD : require.js

CMD : sea.js

基于AMD / CMD思想实现模块之间的依赖和调用



部署上线的时候在基于部署工具（grunt / glup / fis...）最后按照模块打包到同一个页面中  => 后期控制的是模块的显示隐藏



问题：把所有模块都放到一个页面中，模块之间的切换处理起来也不是特别方便，最主要的是第一次加载的时候，隐藏的模块有需要渲染和解析的，导致第一次加载速度过慢的问题

import Vue from 'vue';

import VueRouter from 'vue-router';

import Home from './pages/Home';

import Custom from './pages/Custem';

import System from './pages/System';

import CustomList from './pages/custom/CustomList';

import CustomHandle from './pages/custom/CustomHandle';



Vue.use(VueRouter);

//创建路由配置表

export default new VueRouter({

    //设置路由模式 hash / history

    mode: "hash",

    //配置一级路由  //每一个对象都是一个规则  Home：首页 Custom：客户管理（我的客户、全部客户可用一个组件（CustomList），新增客户用一个组件（CustomHandle）

    //系统设置：System

    //渲染同一个组件，但展示的内容不一样，我们进行区分，可进行传参

    routes: [{

        //请求的是一个/（http://127.0.0.1:8080/#/），我们让其重定向到/home

        path: '/',

        redirect: '/home'   // redirect 重定向 英[ˌriːdəˈrekt , ˈriːdərekt]

        //  component: Home  真实项目中很少会让两个地址渲染一个组件

    },

    {

        // hash路径

        path: '/home',   // http://127.0.0.1:8080/#/

        // 渲染的组件

        component: Home

    }, {

        path: '/custom',

        component: Custom,

        //配置某一个一级路由下面的二级路由

        children: [{

            path: '/custom/list',

            component: CustomList

        }, {

            path: 'handle',  //可用简写，但是不能写斜杠了，否则认为是根目录下访问

            component: CustomHandle

        }]

    }, {

        path: '/system',

        component: System

    }，{

        //每一次地址改变，都会到路由表中自上而下逐一匹配

        //匹配到一级成功，再匹配二级...，在某级中没有则跳出来继续向下匹配

        //直到找到一个完全匹配的，则渲染对应的组件（不再向下匹配了）

        //如果上面都匹配不成功

        path:'*',

        redirect:'/'

        // component:Error

    }]

})

query 和 params 的区别

query问号传递的信息会在地址栏中体现出来

    + 比较丑

    + 基于 router-link 或者 this.$router.push 跳转的时候，传递的参数可以在渲染的组件中获取到

    + 在当前地址下刷新，依然可以基于 this.$router.currentRoute.query获取到（地址中有）



params不会在地址栏中显示

    + 相对美观

    + 基于 router-link 或者 this.$router.push 跳转的时候，传递的参数可以在渲染的组件中也可以获取到

    + 在当前地址下刷新浏览器，params中的信息就清空了

路由中的组件渲染问题

每一次路由的跳转和切换，都是把之前渲染的组件销毁（destoryed），重新渲染新的组件（控制台输出顺序：（BEFORE-CREATE => DATA => CREATE => BEFORE-MOUNT => MOUNTED）

export default {

    console.log("BEFORE-CREATE");

  },

  created() {

    console.log("CREATE");

  },

  beforeMount() {

    console.log("BEFORE-MOUNT");

  },

  mounted() {

    console.log("MOUNTED");

  },

  beforeDestroy() {

    console.log("BEFORE-DESTROY");

  },

  destroyed() {

    console.log("DESTROYED");

  },

  data() {

    console.log("DATA");

    return {};

  },

与路由有关的生命周期函数（导航守卫：路由权限校验）

全局beforeEach => 路由独有beforeEnter => 组件独有beforeRouterEnter => 全局beforeReslove（当前被激活的组件开始渲染） => 全局afterEach => beforeCreate => data => ... （vue的一个生命周期）

“导航”表示路由正在发生改变

组件独有beforeRouterLeave => 全局beforeEach => 全局beforeReslove => 全局afterEach => before-destory => destoryed（vue的一个生命周期）

完整的导航解析流程

1、导航被触发 （浏览器地址改变）

2、在失活的组件里调用离开守卫      （beforeRouterLeave）

3、调用全局的 beforeEach  守卫 

4、在重用的组件里调用 beforeRouterUpdate 守卫（2.2+） （从当前组件又跳到该组件）

5、在路由配置里调用 beforeEnter    （到新组件）

6、解析异步路由组件

7、在被激活的组件里调用 beforeRouteEnter

8、调用全局的 beforeReslove 守卫（2.5+）

9、导航被确认

10、调用全局的 afterEach 钩子

11、触发 DOM 更新

12、用创建好的实例调用 beforeRouterEnter 守卫中传给 next 的回调函数
